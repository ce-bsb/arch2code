# `.arch/` — the pipeline's working directory and audit trail

This directory is where the arch2code pipeline reads its inputs and writes everything it
produces. Nothing here is hand-authored source: with the single exception of this README and
the fixtures under `intake/inbox/`, every file was written by one of the six custom modes
during a run.

Two things follow from that, and they matter more than they look:

1. **These files are evidence, not documentation.** They record what actually happened on a
   given date, with a given model, from a given drawing. They are kept so that a decision made
   in stage 4 can be traced back to a pixel in stage 1.
2. **They are therefore not edited after the fact** — not to fix them, not to improve them,
   and (see the language note below) not to translate them.

---

## Lifecycle: what each subdirectory holds

A run is identified by a `run-id` of the form `YYYYMMDD-HHMM-<short-name>`, and the same
`run-id` names a directory in every stage it reached. You can read a run vertically by
following that id down the tree.

| Path | Written by | Contents |
|---|---|---|
| `intake/inbox/` | humans | Drop zone. Diagrams waiting to be picked up, plus the two synthetic fixtures used by the test suite. Not a run. |
| `intake/<run-id>/` | stage 1 — `arch-intake` | The captured source artifact, copied verbatim out of the inbox. This is the origin of everything downstream. |
| `air/<run-id>/air.json` | stage 2 — `arch-analyst` | The **AIR** (Architecture Intermediate Representation): the contract. Components, connections, boundaries, assumptions, unknowns, and the provenance of the extraction. Every later stage reads this and nothing else. |
| `review/<run-id>/verdict.md` | stage 3 — `arch-critic` | The review report on the AIR, and the gate decision that either opens or blocks stage 4. |
| `build/<run-id>/` | stage 4 — `arch-scaffold` | Generated scaffolding — agent definitions, tools, knowledge bases, and a deploy README. Code, but code that is downstream of the contract. |
| `run/<run-id>/` | stage 5 — `arch-validator` | Experiment execution: `harness.log` and the validation report. Currently empty — neither historical run reached stage 5. |

The gates between stages are blocking. A run that appears in `air/` but not in `build/` did not
fail to be worth building; it was stopped, and the reason is in `review/`.

## Language: these artifacts are in Portuguese on purpose

The harness — modes, rules, skills, scripts, and documentation — has been migrated to English.
**The run artifacts under `air/`, `review/`, `build/`, and `intake/<run-id>/` have not been, and
will not be.**

This is a deliberate decision, not an oversight or a pending task. These files are the faithful
record of executions that took place before the migration, when the pipeline's own language rule
told every mode to write prose in Portuguese. Rewriting them in English would produce documents
that read as if they had been generated by a pipeline configuration that did not exist on the day
they were produced. That is not a translation; it is a retroactive edit of an audit trail for a
regulated client.

So the descriptions, responsibilities, assumptions, verdict prose, and generated file names
(`agente_disputa.yaml`, `regras_cartoes_rag.yaml`, `lista_compras.py`, and the rest) stay exactly
as the runs emitted them. This README is the English entry point that sits beside them. Future
runs, executed against the migrated harness, will produce English artifacts — and the two
generations will be distinguishable by date and by language, which is a feature.
