# `.arch/` — the pipeline's working directory and audit trail

This directory is where the arch2code pipeline reads its inputs and writes everything it
produces. Nothing here is hand-authored source: with the single exception of this README and
the fixtures under `intake/inbox/`, every file was written by one of the six custom modes
during a run.

Two things follow from that, and they matter more than they look:

1. **These files are evidence, not documentation.** They record what actually happened on a
   given date, with a given model, from a given drawing. They are kept so that a decision made
   in stage 4 can be traced back to a pixel in stage 1.
2. **They are therefore not edited after the fact** — not to fix them, not to improve them,
   and (see the language note below) not to translate them.

---

## Lifecycle: what each subdirectory holds

A run is identified by a `run-id` of the form `YYYYMMDD-HHMM-<short-name>`, and the same
`run-id` names a directory in every stage it reached. You can read a run vertically by
following that id down the tree.

| Path | Written by | Contents |
|---|---|---|
| `intake/inbox/` | humans | Drop zone. Diagrams waiting to be picked up, plus the two synthetic fixtures used by the test suite. Not a run. |
| `intake/<run-id>/` | stage 1 — `arch-intake` | The captured source artifact, copied verbatim out of the inbox. This is the origin of everything downstream. |
| `air/<run-id>/air.json` | stage 2 — `arch-analyst` | The **AIR** (Architecture Intermediate Representation): the contract. Components, connections, boundaries, assumptions, unknowns, and the provenance of the extraction. Every later stage reads this and nothing else. |
| `review/<run-id>/verdict.md` | stage 3 — `arch-critic` | The review report on the AIR, and the gate decision that either opens or blocks stage 4. |
| `build/<run-id>/` | stage 4 — `arch-scaffold` | Generated scaffolding — agent definitions, tools, knowledge bases, and a deploy README. Code, but code that is downstream of the contract. |
| `run/<run-id>/` | stage 5 — `arch-validator` | Experiment execution: `harness.log` and the validation report. Currently empty — neither historical run reached stage 5. |

The gates between stages are blocking. A run that appears in `air/` but not in `build/` did not
fail to be worth building; it was stopped, and the reason is in `review/`.

---

## Language: these artifacts are in Portuguese on purpose

The harness — modes, rules, skills, scripts, and documentation — has been migrated to English.
**The run artifacts under `air/`, `review/`, `build/`, and `intake/<run-id>/` have not been, and
will not be.**

This is a deliberate decision, not an oversight or a pending task. These files are the faithful
record of executions that took place before the migration, when the pipeline's own language rule
told every mode to write prose in Portuguese. Rewriting them in English would produce documents
that read as if they had been generated by a pipeline configuration that did not exist on the day
they were produced. That is not a translation; it is a retroactive edit of an audit trail for a
regulated client.

So the descriptions, responsibilities, assumptions, verdict prose, and generated file names
(`agente_disputa.yaml`, `regras_cartoes_rag.yaml`, `lista_compras.py`, and the rest) stay exactly
as the runs emitted them. This README is the English entry point that sits beside them. Future
runs, executed against the migrated harness, will produce English artifacts — and the two
generations will be distinguishable by date and by language, which is a feature.

## Run: `20260716-1548-uci`

**Source diagram:** `intake/20260716-1548-uci/arquitetura_teste_uci.png` (281 KB PNG, no
structured source available — the extraction went down the `vision` path).

### What the AIR captured

A multi-agent investment recommendation architecture on watsonx.orchestrate. Ten components,
ten connections, one boundary, four assumptions, zero unknowns.

The chain is: `AI CHAT` → `Portfolio Agent` → `Recommendation Agent` → `Notification Agent` →
back to `AI CHAT`. The Portfolio Agent reads `Investment Attributes`; the Recommendation Agent
queries a native knowledge base (`Correlating Client/Investment Profiles`) over RAG, calls a
model deployed on watsonx.ai through a tool placeholder, and writes to
`Results Recommendations`; the Notification Agent reads those results and renders the summary.
`External Sources` feeds the knowledge base asynchronously. Everything except the deployed model
and the external sources sits inside the `watsonx.orchestrate` boundary.

Four of the ten connections carry `verified: true` from a second-pass vision check; the other
six are marked `inferred: true` with a note explaining the inference — mostly co-location in the
drawing implying data access. The AIR is at version 2: the changelog records that v1 was blocked
and revised with direct input from the diagram's author.

### The critic's verdict

`review/20260716-1548-uci/verdict.md` — **blocked at v1, approved at v2.**

Three critical findings were raised and all three were resolved by going back to the human who
drew the diagram, rather than by guessing:

- **[C1]** The Notification Agent had no return path to the user. The author confirmed the
  `SUMMARY` connection back to `AI CHAT`; it was added as `inferred`.
- **[C2]** The datastores were drawn adjacent to the agents with no arrows. Four inferred
  read/write connections were added, each with a note stating that the evidence is co-location.
- **[C3]** `recommendation_model` was reclassified from a vague box to `kind: service`,
  `tech: watsonx.ai deployed model`, reached through an explicit tool placeholder with a
  `PLACEHOLDER` endpoint. The separate `rag` component was dropped in favour of the agent's
  native knowledge base, and the model and external sources were moved outside the boundary.

The gate was then opened for stage 4, with three carry-forward items: keep the model tool as a
placeholder until credentials exist, leave the datastore technology abstract, and emit the
`inferred: true` connections as TODO comments rather than as fact.

### What stage 4 generated

`build/20260716-1548-uci/` — 6 files:

```
README.md                                          deploy order + pending TODO table
tools/recommendation_model_tool.py                 @tool stub; raises rather than faking a call
knowledge_bases/correlating_client_investment_profiles.yaml
agents/portfolio_agent.yaml
agents/recommendation_agent.yaml
agents/notification_agent.yaml
agents/multi_agent_orchestration_workflow.yaml     root orchestrator, imported last
```

The build README documents an import order that is mandatory rather than stylistic — collaborator
agents must exist before the orchestrator that references them — and closes with a table of five
open TODOs mapped to owners (ML, Ops, Data, Eng, Architecture). The tool stub keeps
`WATSONX_AI_DEPLOYMENT_URL = "PLACEHOLDER …"` and fails loudly if invoked, which is the intended
behaviour: a loud failure is honest, a silent stub lies about how complete the prototype is.

## Run: `20260721-1129-atendente`

**Source diagram:** `intake/20260721-1129-atendente/arquitetura_agentes_adk_orchestrate.png`
(151 KB PNG, again `vision` — no `.drawio` existed).

### What the AIR captured

A customer-service multi-agent architecture for credit cards. Six components, eight connections,
two boundaries, five assumptions, zero unknowns.

`Agente Atendente Supervisor` is the entry point and router. It dispatches to three sub-agents —
`Agente Disputa` (dispute lifecycle, four tools), `Agente Transbordo` (sentiment, summary, human
handoff, three tools), and `Assistente - Localiza Agência` (branch finder, modelled as an
external watsonx Assistant) — and queries the `Regras Cartões RAG` knowledge base. Three of the
four dispute tools reach `Incidentes (ServiceNow)` over REST; those calls were drawn as dashed
arrows and are captured as three separate connections so the ServiceNow coupling is visible
per tool rather than collapsed into one edge.

All eight connections are `verified: true` at confidence 1.0 after the second pass. The five
assumptions cover exactly the places where the drawing was silent: which system `Lista Compras`
actually talks to, whether `Transbordo` needs an external queue, the missing `app_id` for the
watsonx Assistant, the synchronous nature of intra-platform calls, and the missing ServiceNow
credentials.

### The critic's verdict

`review/20260721-1129-atendente/verdict.md` — **approved on the first pass**, no blocking items,
gate opened for scaffolding.

Four observations were recorded, none of them blocking, each already tracked as an assumption in
the AIR: `Lista Compras` has a different icon and no dashed arrow to ServiceNow (**[O1]**);
`Transbordo` may need an external contact-queue system the drawing does not name (**[O2]**);
the ServiceNow authentication is nowhere in the diagram (**[O3]**); and the watsonx Assistant
`app_id` and URL are absent (**[O4]**). The pattern worth noticing is that the critic did not
invent any of these four — it named them, pointed at the assumption entry that carries them, and
told the scaffold to emit placeholders instead of plausible values.

### What stage 4 generated

`build/20260721-1129-atendente/` — 9 files:

```
README.md                                   connections → tools → KB → agents (leaves before root)
tools/lista_compras.py                      stub, deliberately with no ServiceNow connection
tools/servicenow_tools.py                   lista_disputa, abre_disputa, consulta_disputa_por_id
tools/transbordo_tools.py                   sentimento, resumo, transbordo
knowledge_bases/regras_cartoes_rag.yaml     documents section commented out, TODO to populate
agents/agente_disputa.yaml
agents/agente_transbordo.yaml
agents/assistente_localiza_agencia.yaml     external watsonx Assistant, api_url + app_id PLACEHOLDER
agents/agente_atendente_supervisor.yaml     root orchestrator, imported last
```

Each of the four uncertainties the critic flagged is visible in the output as a `PLACEHOLDER`
string or a `TODO`, and the build README's closing table lists all six with file paths and an
impact rating. Nothing was quietly filled in.

## Known issues with these runs

Both runs predate the current AIR schema and the current validator. They are the **baseline to be
beaten, not the example to copy.** Read them for how the stages hand off to each other and for
how the critic handled uncertainty — not as templates for a well-formed AIR.

Reproduce the following with:

```bash
python3 .bob/skills/air-normalizer/scripts/validate_air.py .arch/air/<run-id>/air.json --gate
```

- **Both AIRs fail `validate_air.py --gate` with exit code 1.** `20260716-1548-uci` reports
  4 errors / 2 warnings; `20260721-1129-atendente` reports 4 errors / 1 warning. Neither would
  pass the gate as it stands today.
- **Neither AIR has a `meta{}` object.** They carry `run_id`, `source_artifact`,
  `extraction_path`, and `overall_confidence` as top-level keys, which is where the pre-schema
  shape put them. The validator looks them up under `meta`, so it prints `run_id: ?`,
  `source: ?`, `path: ?`, and — the consequential one — reads `overall_confidence` as `0.00` and
  raises `below the minimum of 0.75` on both. The confidence recorded in the files is actually
  0.85 and 0.90. The error is real (the field is not where the contract says it is), but the
  number in it is an artefact of the lookup, not a measurement.
- **No component or connection carries an `evidence` field.** Zero occurrences of the string in
  either file. There is no `bbox`, no `cell`, no `label_text` — nothing that ties a component
  back to a region of the source PNG. Provenance survives only at the whole-file level, in
  `_provenance`. This is the single biggest gap relative to the current contract: the audit trail
  stops at "this run read that image", and cannot go down to "this box, at these coordinates".
- **`experiment_plan` is `null` in both** — the key is simply absent. That trips two of the four
  errors on each run: no `hypotheses` and no `out_of_scope`. In the terms the pipeline uses, both
  runs produced a demo rather than an experiment: there is no falsifiable statement anywhere in
  either AIR, and nothing declares what will not be built. That is also why stage 5 never ran and
  `run/` is empty.
- **`unknowns[]` is empty in both.** Every gap that survived extraction was written into
  `assumptions[]` instead, each with an `impact` — so nothing was silently swallowed, and the
  assumptions are genuinely good. But an assumption is a decision the pipeline made and moved
  past, while an unknown is a question that stops it and goes to the human. Two vision
  extractions from hand-drawn architectures producing zero open questions between them is a
  calibration signal, not a clean bill of health.
- **Neither `verdict.md` contains the gate string.** `grep` for `VERDICT:` (or its pre-migration
  Portuguese form) returns nothing in either file. Both express the decision in prose — a
  `## Decisão de portão` section reading `**Portão ABERTO.**` — which a human reads correctly and
  the stage-4 precondition, which looks for a literal substring, does not. Stage 4 ran on both
  runs anyway, which means the gate was satisfied by a person, not by the mechanism. A gate that
  is only enforced when someone happens to be watching is not a gate.
- **Run-specific defects.** `20260716-1548-uci` contains a synchronous cycle
  (`ai_chat → portfolio_agent → recommendation_agent → notification_agent → ai_chat`) that the
  validator blocks on as circular coupling, and `multi_agent_orchestration_workflow` is an orphan
  component with no connection at all — usually an arrow lost in extraction, though here it is
  more likely that platform-native orchestration genuinely has no drawn edges.
  `20260721-1129-atendente` has a broken reference: connection
  `localiza_agencia_to_watsonx_assistant` points at `to: "watsonx_assistant_external"`, an id
  that does not exist in `components[]`.

None of the above is being fixed in place, for the reason given in the language note: these are
records of what happened. The way to improve on them is to execute a new run against the current
harness and compare.
